# 模块组件详解

> 本文档详细说明每个模块的用途、核心组件、实现原理和设计思路。

---

## 一、pandora-common — 公共基础模块

### 1.1 定位

全项目最底层的模块，所有其他模块都依赖它。只包含**纯 Java 类型**，不依赖 Spring 框架，保证零侵入。

### 1.2 核心组件

| 组件 | 路径 | 用途 |
|------|------|------|
| `Result<T>` | common/result/ | 统一响应封装。所有 API 返回格式统一为 `{code, message, data, timestamp}` |
| `ResultCode` | common/result/ | 错误码枚举（200 成功, 400 请求错误, 401 未授权, 403 无权限, 500 内部错误等） |
| `PageResult<T>` | common/result/ | 分页响应封装（list + total + pageNum + pageSize） |
| `PageQuery` | common/base/ | 分页查询基类（pageNum, pageSize, orderBy） |
| `BaseEntity` | common/base/ | 实体基类（id, createTime, updateTime） |
| `BusinessException` | common/exception/ | 业务异常（可预期的错误，如"用户不存在"） |
| `SystemException` | common/exception/ | 系统异常（不可预期的错误） |
| `LoginUser` | common/security/ | 当前登录用户上下文（userId, username, roles, permissions, userType, dataScope） |
| `LoginUserHolder` | common/security/ | ThreadLocal 持有者，在请求线程中随时获取当前用户信息 |
| `UserType` | common/security/ | 用户类型枚举（B_USER=管理员, C_USER=消费者） |
| `DataScopeType` | common/security/ | 数据权限范围枚举（ALL, DEPT_AND_CHILDREN, DEPT, SELF, CUSTOM） |
| `TokenConstants` | common/security/ | Token 生命周期管理的 Redis Key 常量和 JWT Claim 名称 |
| `@RequiresPermission` | common/annotation/ | 声明式权限注解，标注在 Controller 方法上表示需要指定权限 |
| `@RequiresUserType` | common/annotation/ | 声明式用户类型注解，限制 B/C 端访问（支持类级和方法级） |
| `@DataPermission` | common/annotation/ | 声明式数据权限注解，标注在 Mapper 方法上触发数据过滤 |
| 工具类 | common/util/ | JsonUtils, DateUtils, BeanCopyUtils, IdGenerator, SecurityUtils, AssertUtils, ServletUtils |

### 1.3 设计思路

- **零依赖原则：** 不依赖 Spring，不依赖 MyBatis，不依赖任何框架。任何模块都可以安全引入。
- **ThreadLocal 模式：** `LoginUserHolder` 使用 ThreadLocal 存储当前用户，在请求线程中任何地方都可以通过 `LoginUserHolder.get()` 获取，无需从 Controller 层层传递。Filter 在请求结束时自动清理，防止内存泄漏。

## 二、pandora-domain — 领域层

### 2.1 定位

DDD 的核心层，定义业务领域的实体、值对象、聚合根、领域事件和仓储接口。**只定义"是什么"和"做什么"，不关心"怎么做"。**

### 2.2 核心组件

| 组件 | 用途 |
|------|------|
| `Entity` | 实体基类接口，有唯一标识 |
| `ValueObject` | 值对象标记接口，无唯一标识，通过属性值判断相等 |
| `AggregateRoot` | 聚合根基类，领域事件的发布者 |
| `Repository` | 仓储接口，定义持久化契约（增删改查） |
| `DomainEvent` | 领域事件基类 |
| `DomainEventPublisher` | 领域事件发布器接口 |
| `User` | 用户聚合根 |
| `UserId` | 用户 ID 值对象 |
| `UserStatus` | 用户状态枚举 |
| `UserRepository` | 用户仓储接口 |
| `UserCreatedEvent` | 用户创建领域事件 |

### 2.3 设计思路

- **纯净领域：** 不引入任何框架注解（没有 `@Entity`、`@TableName`），保持业务逻辑的纯净表达。
- **仓储接口在领域层定义，实现在基础设施层：** 这是 DDD 的"依赖倒置"核心——领域层不知道数据存在 MySQL 还是 MongoDB 中。
- **领域事件解耦：** User 聚合根在创建时发布 `UserCreatedEvent`，由基础设施层的事件监听器处理后续逻辑（如发送欢迎邮件）。

## 三、pandora-application — 应用层

### 3.1 定位

协调领域对象完成用例（Use Case）。不包含业务规则，只做**编排和转换**。

### 3.2 核心组件

| 组件 | 用途 |
|------|------|
| `UserApplicationService` | 用户用例编排（创建用户、修改密码、查询用户） |
| `CreateUserCommand` | 创建用户命令（CQRS Command） |
| `ChangePasswordCommand` | 修改密码命令 |
| `UserDTO` | 用户数据传输对象（面向外部的视图） |
| `UserAssembler` | 领域对象 ↔ DTO 转换器 |
| `AiService` | AI 服务接口 |
| `FileService` | 文件服务接口 |

### 3.3 设计思路

- **CQRS 分离：** 写操作用 Command 对象封装（CreateUserCommand），读操作直接返回 DTO。
- **Assembler 转换：** 领域对象（User）和 DTO（UserDTO）之间通过 Assembler 转换，避免领域对象直接暴露给外部。
- **接口隔离：** AiService 和 FileService 只定义接口，具体实现由基础设施层提供。

## 四、pandora-infrastructure — 基础设施层

### 4.1 定位

DDD 中最"脏"的层——所有框架集成、数据库访问、外部服务调用、安全配置都在这里。它**实现**领域层和应用层定义的接口。

### 4.2 持久化模块

| 组件 | 用途 |
|------|------|
| `UserDO` + `UserMapper` | MyBatis-Plus 用户表 ORM |
| `UserJpaEntity` + `UserJpaRepository` | JPA 用户表 ORM（双 ORM 共存演示） |
| `UserRepositoryImpl` | 实现领域层的 `UserRepository` 接口 |
| `UserConverter` | DO ↔ 领域对象转换（MapStruct） |
| `BaseDO` | MyBatis-Plus 实体基类（id, createBy, updateBy, createTime, updateTime, deleted） |
| `MyMetaObjectHandler` | 自动填充创建时间/更新时间 |

**RBAC 持久化（sys_* 系列）：**

| 组件 | 对应表 |
|------|--------|
| `SysDeptDO` + `SysDeptMapper` | sys_dept（部门） |
| `SysRoleDO` + `SysRoleMapper` | sys_role（角色） |
| `SysMenuDO` + `SysMenuMapper` | sys_menu（菜单/权限） |
| `SysUserRoleDO` + `SysUserRoleMapper` | sys_user_role（用户-角色关联） |
| `SysRoleMenuDO` + `SysRoleMenuMapper` | sys_role_menu（角色-菜单关联） |
| `SysRoleDeptDO` + `SysRoleDeptMapper` | sys_role_dept（角色-部门关联） |

### 4.3 安全模块

| 组件 | 用途 | 实现原理 |
|------|------|----------|
| `SecurityConfig` | Spring Security 配置 | 4 条有序 FilterChain（公共/B端/C端/兜底），每条链配置不同的 JWT 转换器 |
| `LoginUserFilter` | 登录用户上下文注入 | 位于 BearerTokenAuthenticationFilter 之后，从 JWT 构建 LoginUser 并存入 ThreadLocal |
| `TokenBlacklistService` | Token 黑名单校验 | 检查 Redis 中 `token:blacklist:{jti}` 是否存在 + Token 代次是否过期 |
| `PermissionAspect` | AOP 权限校验 | 拦截 @RequiresPermission 和 @RequiresUserType 注解，校验当前用户是否具有权限 |
| `PermissionService` | SpEL 权限表达式 | 提供 hasPermission/hasRole/isBUser/isCUser 方法给 @PreAuthorize 使用 |
| `PermissionCacheService` | 权限缓存 | 用户权限数据缓存到 Redis，减少数据库查询。使用 CompletableFuture 异步加载 |

**数据权限模块：**

| 组件 | 用途 |
|------|------|
| `DataScopeStrategy` | 策略接口——根据数据范围类型生成 SQL 条件 |
| `AllDataScopeStrategy` | 全部数据——不追加条件 |
| `DeptDataScopeStrategy` | 本部门——`WHERE dept_id = ?` |
| `DeptAndChildrenStrategy` | 本部门及子部门——`WHERE dept_id IN (...)` |
| `SelfDataScopeStrategy` | 仅本人——`WHERE create_by = ?` |
| `CustomDataScopeStrategy` | 自定义——`WHERE dept_id IN (角色关联部门)` |
| `DataScopeStrategyFactory` | 工厂——根据 DataScopeType 获取对应策略 |
| `DataPermissionInterceptor` | MyBatis-Plus 拦截器——在 SQL 执行前动态追加数据权限条件 |

### 4.4 配置模块

| 组件 | 用途 |
|------|------|
| `RedisConfig` | Redis 序列化配置（Key=String, Value=JSON） |
| `MybatisPlusConfig` | MyBatis-Plus 分页插件 + Mapper 扫描 + 数据权限拦截器注册 |
| `JpaConfig` | JPA 配置 |
| `MinioConfig` | MinIO 客户端配置 |
| `AiAutoConfiguration` | Spring AI 条件加载（类路径有 ChatClient 时才注册） |

### 4.5 RBAC 服务

| 组件 | 用途 | 设计模式 |
|------|------|----------|
| `SysDeptService` | 部门增删改查 + 树形结构构建 | 模板方法（继承 BaseCrudService） |
| `SysRoleService` | 角色管理 + 角色权限分配 | 模板方法 |
| `SysMenuService` | 菜单管理 + 权限标识管理 + 树形结构 | 模板方法 |

### 4.6 其他

| 组件 | 用途 |
|------|------|
| `MinioService` / `MinioFileService` | MinIO 文件操作（上传/下载/删除/预签名URL） |
| `AiChatService` / `AiServiceFallback` | AI 聊天服务（有 Spring AI 时使用真实实现，否则降级） |
| `SpringDomainEventPublisher` | 领域事件发布器实现（基于 Spring ApplicationEventPublisher） |
| `UserEventHandler` | 用户领域事件监听器 |

## 五、pandora-api — 接口层

### 5.1 定位

对外暴露 REST API。负责请求接收、参数校验、调用应用层服务、返回统一响应。

### 5.2 核心组件

| 组件 | 路径前缀 | 用途 |
|------|----------|------|
| `UserController` | /api/users | 用户 CRUD（DDD 风格） |
| `SysRoleController` | /api/system/role | 角色管理（B端，@RequiresPermission） |
| `SysMenuController` | /api/system/menu | 菜单管理（B端） |
| `SysDeptController` | /api/system/dept | 部门管理（B端） |
| `AppMemberController` | /api/app/member | 会员功能（C端，@RequiresUserType(C_USER)） |
| `FileController` | /api/files | 文件上传下载（MinIO） |
| `AiController` | /api/ai | AI 对话接口 |
| `AuthProxyController` | /api/auth/token | OAuth2 Token 代理（给 Swagger 用） |
| `BaseCrudController` | — | CRUD 控制器基类（泛型化 list/get/create/update/delete） |
| `GlobalExceptionHandler` | — | 全局异常处理（BusinessException→400, 其他→500） |
| `SwaggerConfig` | — | OpenAPI 3.0 文档配置（B端OAuth2 + C端Bearer） |

### 5.3 API 路径约定

```
/api/auth/**      → 认证相关（登录/登出/刷新），公开访问
/api/admin/**      → B端管理接口，需要 B端Token + RBAC 权限
/api/system/**     → 系统管理接口（角色/菜单/部门），需要 B端Token + RBAC
/api/app/**        → C端消费者接口，需要 C端Token
/api/public/**     → 公开接口，无需认证
/api/users/**      → 用户 CRUD（DDD 演示）
/api/files/**      → 文件操作
/api/ai/**         → AI 接口
```

## 六、pandora-auth — 认证授权服务

### 6.1 定位

独立的认证服务，负责用户身份验证和 JWT Token 签发。可单独部署（端口 9100），也可通过 Auth 的 API 被网关代理。

### 6.2 核心组件

**认证策略（策略模式）：**

| Provider | 登录类型 | 说明 |
|----------|----------|------|
| `UsernamePasswordAuthProvider` | USERNAME_PASSWORD | 用户名密码认证（BCrypt） |
| `SmsCodeAuthProvider` | SMS_CODE | 短信验证码认证 |
| `WechatMiniProgramAuthProvider` | WECHAT_MINI_PROGRAM | 微信小程序 code2session |

`AuthenticationService` 维护一个 `Map<LoginType, AuthProvider>`，根据 `loginType` 字段自动路由到对应的认证提供者。

**Token 服务：**

| 方法 | 说明 |
|------|------|
| `generateBToken()` | 生成 B端 JWT（含 roles/deptId/dataScope/tgen，2h 有效期） |
| `generateCToken()` | 生成 C端 JWT（含 nickname/phone/memberLevel/tgen，7d 有效期） |
| `logout()` | AccessToken 加入黑名单 + RefreshToken 从白名单删除 |
| `forceLogoutUser()` | 递增用户 Token 代次，使所有已签发 Token 失效 |
| `isRefreshTokenValid()` | 检查 RefreshToken 是否仍在 Redis 白名单中 |
| `consumeRefreshToken()` | 消费 RefreshToken（一次性使用） |

**B/C 端分离：**

| 组件 | 服务对象 |
|------|----------|
| `AuthUserService` | B端用户（查 sys_user 表） |
| `AuthMemberService` | C端会员（查 sys_member 表） |
| `AuthenticationService` | B端认证编排 |
| `MemberAuthService` | C端认证编排 |

**API 端点：**

| 端点 | 方法 | 说明 |
|------|------|------|
| `/auth/admin/login` | POST | B端登录 |
| `/auth/app/login` | POST | C端登录 |
| `/auth/logout` | POST | 登出（需 Authorization 头） |
| `/auth/refresh` | POST | 刷新 Token（自动识别 B/C 端） |
| `/auth/sms/send` | POST | 发送短信验证码 |
| `/auth/admin/force-logout/{userId}` | POST | 强制踢出 B端用户 |
| `/auth/admin/force-logout/member/{memberId}` | POST | 强制踢出 C端会员 |

### 6.3 密码安全

- 存储：BCrypt 加密（`$2a$10$...`）
- 迁移：`PasswordMigrationRunner` 在启动时自动将明文密码加密
- 校验：`BCryptPasswordEncoder.matches(raw, encoded)`

## 七、pandora-start — 单体启动模块

### 7.1 定位

单体部署的入口模块。通过 Maven 依赖聚合所有其他模块，配置数据源、Redis、MinIO 等基础设施。

### 7.2 关键配置

- `application.yml` — 生产配置（MySQL 主从、Redis、MinIO、安全密钥）
- `application-dev.yml` — 开发环境覆盖配置
- `application-h2.yml` — 内存数据库配置（快速启动/演示用）
- `PandoraApplication.java` — Spring Boot 启动类

### 7.3 运行

```bash
mvn spring-boot:run -pl pandora-start -DskipTests              # 默认配置
mvn spring-boot:run -pl pandora-start -DskipTests -Dspring.profiles.active=dev  # 开发模式
```

## 八、pandora-gateway — 微服务网关

### 8.1 定位

微服务模式的统一入口。负责路由转发、Token 验签、限流熔断。

### 8.2 核心组件

| 组件 | 用途 |
|------|------|
| `GatewayApplication` | 网关启动类 |
| `AuthGlobalFilter` | 全局认证过滤器——验证 JWT 有效性，将用户信息传递给下游服务 |

### 8.3 路由规则

```yaml
routes:
  - id: auth-service
    uri: lb://pandora-auth-server
    predicates:
      - Path=/auth/**
  - id: user-service
    uri: lb://pandora-service-user
    predicates:
      - Path=/api/**
```

## 十、pandora-service — 微服务业务模块

### 10.1 定位

微服务模式下的业务服务聚合 POM。每个子模块是一个独立可部署的微服务。

### 10.2 pandora-service-user

| 组件 | 用途 |
|------|------|
| `UserServiceApplication` | 用户服务启动类（微服务模式独立部署） |

依赖 `pandora-api`、`pandora-infrastructure`、`pandora-common-cloud`，注册到 Nacos。
