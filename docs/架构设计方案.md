# pandora-arch — 架构设计方案

> 工程：`pandora-arch` · GroupId：`cn.pandora` · 版本：`1.0.0`
> 基线：JDK 25 + Spring Boot 4.0.5 + Spring Cloud Alibaba 2025.1.0.0 + DDD
> 更新日期：2026-04

---

## 一、架构定位

`pandora-arch` 是一套**企业级代码底座**，基于 **Spring Boot 4.0 + DDD（领域驱动设计）+ OAuth2** 构建，目标是提供一个开箱即用的脚手架，使开发团队只需关注业务逻辑，不需要重复搭建认证、权限、安全、存储等基础设施。

**核心特性：**

- 单体与微服务双模式部署，同一份代码无需修改
- B 端（管理后台）与 C 端（消费者 / 会员）用户完全隔离
- 企业级 RBAC（角色 / 菜单 / 按钮 / API 权限）+ 数据权限隔离
- JWT Token 完整生命周期管理（登出失效 + 强制踢人）
- 多数据源动态切换 + **MyBatis-Flex / Spring Data JPA 双 ORM 共存**
- 多渠道认证（用户名密码 / 短信验证码 / 微信 / QQ / 企业微信 等）
- **Redis 只使用 Redisson**（禁用 Lettuce/Jedis），内置分布式锁 / 限流 / 布隆
- Kafka（可选）+ MongoDB（可选）+ MinIO + Spring AI（可选）

## 三、DDD 四层架构

```
┌──────────────────────────────────────────────────────────────┐
│                   接口层 (pandora-api)                         │
│  REST Controller / Swagger / GlobalExceptionHandler           │
├──────────────────────────────────────────────────────────────┤
│                  应用层 (pandora-application)                  │
│  Application Service / DTO / Command / Assembler              │
├──────────────────────────────────────────────────────────────┤
│                  领域层 (pandora-domain)                       │
│  Entity / ValueObject / AggregateRoot / Repository            │
│  DomainEvent / DomainService                                  │
├──────────────────────────────────────────────────────────────┤
│               基础设施层 (pandora-infrastructure)               │
│  Persistence / Security / Config / Cache / Lock / MQ          │
│  MyBatis-Flex Mapper / JPA Repository / Redisson / MinIO      │
│  Spring AI / MongoDB / Kafka                                  │
└──────────────────────────────────────────────────────────────┘

横切关注点：
  pandora-common        → 所有层共享的基础类型（Result / 异常 / 工具 / 注解 / 安全上下文）
  pandora-common-cloud  → 微服务公共配置（Nacos / Sentinel，仅微服务模块引入）
  pandora-auth          → 独立的 OAuth2 认证授权服务
```

**依赖方向（严格单向）：** 接口层 → 应用层 → 领域层 ← 基础设施层

领域层不依赖任何外部框架，保持纯净。基础设施层实现领域层定义的接口。

## 四、安全架构

### 4.1 认证流程

```
用户请求 → pandora-auth :9100
           │
           ├── B 端: /auth/admin/login → AuthenticationService
           │         → UsernamePasswordAuthProvider / SmsCodeAuthProvider / ...
           │         → AuthTokenService.generateBToken()
           │         → JWT (jti + sub + userType=1 + roles + deptId + dataScope + tgen)
           │
           └── C 端: /auth/app/login → MemberAuthService
                     → 短信验证码 / 社交登录
                     → AuthTokenService.generateCToken()
                     → JWT (jti + sub + userType=2 + nickname + phone + memberLevel + tgen)
```

### 4.2 鉴权流程（资源服务器）

```
请求到达 → SecurityFilterChain（按路径匹配）
    │
    ├── /swagger-ui/** /api/auth/** /api/public/** → Chain 1: permitAll
    ├── /api/admin/**  /api/system/**              → Chain 2: JWT + RBAC
    ├── /api/app/**                                → Chain 3: JWT + 身份认证
    └── 其他                                       → Chain 4: JWT 兜底
    │
    ↓ BearerTokenAuthenticationFilter（HMAC 验签）
    ↓ LoginUserFilter
        ├── 黑名单：jti 是否在 Redis 中
        ├── 代次：tgen 是否低于当前代次
        ├── 构建 LoginUser（B 端含完整 RBAC，C 端轻量级）
        └── 写入 ThreadLocal（LoginUserHolder）
    │
    ↓ Controller
        ├── @RequiresPermission("system:role:list") → AOP 校验
        ├── @RequiresUserType(UserType.B_USER)       → AOP 校验
        └── @DataPermission                          → MyBatis-Flex 拦截器追加 SQL
```

### 4.3 Token 生命周期

```
签发                                  失效
 │                                     │
 ├─ AccessToken ──── jti + tgen ─────→ 黑名单 / 代次过期 / 自然过期
 │   B 端有效期：2 小时                 ├─ 主动登出：jti 加入黑名单
 │   C 端有效期：7 天                   └─ 强制踢人：tgen 递增
 │
 └─ RefreshToken ── jti ─────────────→ 白名单删除 / 一次性消费 / 自然过期
     B 端有效期：7 天                   ├─ 主动登出：白名单删除
     C 端有效期：30 天                  └─ 刷新：消费后签发新的
```

### 4.4 数据权限隔离

| 数据范围 | SQL 追加条件 |
|---------|-------------|
| ALL（全部数据） | 不追加条件 |
| DEPT_AND_CHILDREN（本部门及子部门） | `WHERE dept_id IN (本部门 + 所有子部门)` |
| DEPT（本部门） | `WHERE dept_id = {当前用户部门 ID}` |
| SELF（仅本人） | `WHERE create_by = {当前用户 ID}` |
| CUSTOM（自定义） | `WHERE dept_id IN (角色关联的部门列表)` |

实现：策略模式（`DataScopeStrategy`）+ **MyBatis-Flex 标准 `Interceptor`**（`DataPermissionInterceptor`）。

> 注：从 MyBatis-Plus 切换到 MyBatis-Flex 后，拦截器从 `MybatisPlusInterceptor + InnerInterceptor` 改为标准 MyBatis `Interceptor`；自动填充从 `MetaObjectHandler` 改为 `InsertListener` / `UpdateListener`。

## 六、模块依赖关系

```
pandora-start（单体入口）
  ├── pandora-api
  │     └── pandora-application
  │           └── pandora-domain
  ├── pandora-infrastructure
  │     ├── pandora-domain
  │     └── pandora-application
  └── pandora-common

pandora-auth（独立认证服务）
  └── pandora-common

pandora-gateway（微服务网关，唯一可直接引用 spring-cloud-starter-* 的模块）
  ├── pandora-common
  ├── spring-cloud-starter-gateway-server-webflux
  ├── spring-cloud-starter-alibaba-nacos-discovery
  ├── spring-cloud-starter-alibaba-nacos-config
  └── spring-cloud-starter-alibaba-sentinel

pandora-service-user（微服务业务模块）
  ├── pandora-api
  ├── pandora-infrastructure
  └── pandora-common-cloud
        └── （Nacos + Sentinel，禁止 OpenFeign）
```

**零耦合设计原则：**
- `pandora-auth` 不依赖 `pandora-infrastructure`，通过 Redis 共享 Token 黑名单
- `pandora-domain` 不依赖任何 Spring 框架，保持领域纯净
- 模块间通过接口/事件通信，不直接引用实现类
- 除 `pandora-gateway` 外，所有模块严禁直接依赖 `org.springframework.cloud:spring-cloud-starter-*`

## 七、设计模式应用

| 模式 | 应用场景 | 实现位置 |
|------|---------|---------|
| 策略模式 | 数据权限范围计算 | `DataScopeStrategy` + `DataScopeStrategyFactory` |
| 策略模式 | 多渠道认证提供者 | `AuthProvider` + `AuthenticationService` |
| 模板方法 | CRUD 服务基类 | `BaseCrudService` (beforeSave/afterSave 钩子) |
| 观察者 | 领域事件发布订阅 | `DomainEventPublisher` + `@EventListener` |
| 装饰器 | MyBatis-Flex 拦截器增强 SQL | `DataPermissionInterceptor` |
| 工厂 | 数据权限策略工厂 | `DataScopeStrategyFactory` |
| 代理 | AOP 权限校验 | `PermissionAspect` |

## 八、多线程与异步

| 场景 | 技术 | 说明 |
|------|------|------|
| 用户权限/菜单并行加载 | `CompletableFuture` | 登录时并行查询角色、权限、菜单、数据范围 |
| 领域事件异步处理 | `@Async` + `@EventListener` | 用户创建后异步发送通知 |
| Redis 缓存预热 | `CompletableFuture.supplyAsync` | 权限缓存异步加载 |
| 虚拟线程 | `spring.threads.virtual.enabled=true` | Spring Boot 4 + JDK 25 默认支持 |

## 九、单体 vs 微服务

| 维度 | 单体模式 | 微服务模式 |
|------|---------|-----------|
| 入口 | `pandora-start` :8080 | `pandora-gateway` :8888 |
| Auth | `pandora-auth` :9100 | `pandora-auth` :9100 + Nacos 注册 |
| 业务 | 所有模块打包 | `pandora-service-user` 等独立部署 |
| 服务发现 | 不需要 | Nacos 集群 |
| 配置中心 | 本地 `application.yml` | Nacos Config |
| 负载均衡 | 不需要 | Spring Cloud LoadBalancer（仅网关） |
| 熔断限流 | 不需要 | Sentinel |
| 服务间调用 | 直接方法调用 | RestClient + Nacos NamingService / Dubbo（禁用 OpenFeign） |

切换方式：通过 `NACOS_ENABLED` 环境变量控制 Nacos 注册发现开关（`pandora-auth` 默认根据 profile，`pandora-start` 不注册）。
